# TokenSee 功能建议与迭代方向

本文基于当前仓库代码（后端 `/v1` API + 前端 Next 仪表盘页面）对产品能力做一次快速梳理，并给出可扩展的功能建议与优先级，供后续迭代规划参考。

## 现有能力概览（我对项目的理解）

- **后端（API）**：以 `/v1` 为核心的多链数据 API，覆盖交易解码、地址画像/活动、告警（含 SSE）、实体库（entity library）、smart money、intelligence、security、资金流向（flow）、webhooks、stats/price 等。
- **前端（Web）**：Next.js 仪表盘式产品，包含 Dashboard、Alerts、Alert Rules、Entity、Smart Money、Intelligence、Security、Flow、Webhooks、Docs/Playground 等页面。

## 最值得新增的功能（按优先级）

### 1) 可观测性与调试（建议优先做）

- **前端内置“连通性/配置”面板**
  - 展示当前 API 目标（同源代理 `/api/v1` 或直连 `NEXT_PUBLIC_API_URL`）
  - 最近一次请求的错误信息（status / body 摘要）
  - SSE 连接状态、重连次数、最后心跳时间
- **端到端请求追踪**
  - 后端统一返回 `x-request-id`
  - 前端在错误弹窗中展示 `request-id`（便于快速定位后端日志）

价值：显著降低“看起来像后端挂了/其实是网络与配置问题”的排障成本，提升开发与演示稳定性。

### 2) 告警与自动化（更产品化）

- **告警规则增强**
  - 条件支持组合（AND/OR）、按资产/合约过滤、白名单/黑名单实体
  - 冷却时间（cooldown）、每日触发上限
  - 规则命中采样与回放（为什么触发、触发了哪些事件）
- **Webhook 可靠性**
  - 失败重试策略、退避（backoff）、死信队列（DLQ）
  - Webhook 签名（HMAC）与接收方验证
  - 投递记录与重放（replay）

价值：把“看板”升级为“自动化引擎”，用户粘性与付费点更明确。

### 3) 实体库与情报闭环

- **实体关系图谱**
  - 把 `entity / address / flow` 串起来，提供证据链：来源、置信度、最近交互、共同资金路径
  - 图谱节点支持“实体/地址/合约/协议”，边支持“转账/交互/同实体/同集群”
- **用户自定义标签与协作**
  - 自建实体/标签、审核流（proposal → review → merge）
  - 变更历史、审计日志、回滚

价值：沉淀差异化数据资产，形成“数据 → 解释 → 决策 → 反馈”的闭环。

### 4) Security / Intelligence 深化

- **可解释的风险评分（Explainable Risk Score）**
  - 汇总 sanctioned/hacker/mixer exposure、交互路径、频次、金额等生成分数
  - 输出“为什么高风险”的解释卡片与证据
- **时间线与调查报告**
  - 以地址/实体为中心的 7/30/90 天时间线
  - 一键生成调查报告（可导出 Markdown/PDF）

价值：提升专业用户的“调查效率”，增强对外演示效果。

### 5) 工程化与稳定性（中长期）

- **本地开发一键启动**
  - `npm run dev:all` 同启前后端，自动注入端口/代理
  - 统一 `.env` 模板与检查（缺关键变量时给出明确提示）
- **缓存/限流/降级**
  - 昂贵查询 Redis 缓存与分层 TTL
  - 外部 RPC 超时熔断、fallback、数据延迟提示
  - 前端对“部分链不可用”做友好展示（而不是统一报错）

价值：减少环境不稳定导致的演示失败，提高线上稳定性与成本可控性。

## 建议先落地的“高性价比”改动（短平快）

- **A. 前端 Debug 面板 + 错误展示 request-id**
  - 投入小、能立刻提升排障效率
- **B. Webhook：最小可用的签名校验 + 重试/死信**
  - 直接增强可用性与可产品化程度

## 下一步落地方式（推荐节奏）

1. 先补齐“可观测性/调试”与“错误可解释性”（让系统更可控）
2. 再增强“告警与自动化”（从看板走向引擎）
3. 最后用“实体库闭环 + 风险评分”沉淀长期壁垒
